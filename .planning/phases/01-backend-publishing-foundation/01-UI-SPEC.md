## Phase Surface Contract

| Surface | Contract |
|---------|----------|
| `/` marketing page | Preserve current layout, copy hierarchy, language toggle behavior, pilot/login modals, and visual density. No new backend publishing status UI appears on marketing page. |
| `/app` workspace | Preserve existing module navigation and demo workflow. Replace publish-critical browser state with API-backed state without changing the user's main path through campaign input, review, approval, and package/save controls. |
| Approval area | Add a compact approval snapshot panel near the selected platform draft. It must show platform, draft version, approver, approved timestamp, idempotency key suffix, publish target placeholder, caption/body, CTA, media refs, provider payload summary, and disclosure/settings placeholder. |
| Status timeline | Show a per-platform lifecycle timeline for each platform draft: `draft`, `needs_review`, `approved`, `queued`, `publishing`, `published`, `failed`, `retry_needed`, `manual_fallback_required`. Use one stable row or horizontal stepper per platform; do not resize or reorder steps when state changes. |
| Retry affordance | Show retry only for `failed` or `retry_needed` jobs. Label it `Retry publish`. Require an explicit confirmation that a new attempt will be recorded and idempotency will prevent duplicate published outcomes. |
| Hidden diagnostics | Add `/app/debug` as the hidden operator route. Keep it outside the sidebar's normal merchant navigation. It must show jobs, attempts, statuses, trace IDs, error classes, redacted diagnostics, next action, timestamps, approver, and draft version. |
| Security display | Never show secrets, OAuth payloads, tokens, cookies, authorization headers, refresh tokens, app secrets, or raw provider credentials. Use labels such as `Redacted provider diagnostics` and display only safe summaries. |

## Spacing Scale

Declared values (must be multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Segmented-control padding, icon/text gaps, timeline dot-to-label gaps |
| sm | 8px | Status chip padding, compact metadata rows, table cell inner gaps |
| md | 16px | Default panel padding, form/control gaps, timeline row spacing |
| lg | 24px | App page padding, major panel padding, debug route section spacing |
| xl | 32px | Main workspace section gaps, two-column debug layout gaps |
| 2xl | 48px | Desktop route-level spacing only when separating unrelated sections |
| 3xl | 64px | Existing marketing page spacing only; do not use for new app workflow surfaces |

Exceptions:
- Interactive controls must be at least 44px tall on touch viewports and at least 40px tall on dense desktop/operator tables.
- Timeline dots are 12px diameter; connecting lines are 2px.
- Status chips use 8px vertical and 10px horizontal padding.
- New cards/panels use 8px radius for dense operational surfaces. Existing legacy app panels may keep their current radius until intentionally refactored.

Use exactly these four sizes and two weights for new Phase 1 workflow/debug UI. Existing marketing hero typography may remain unchanged.

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Label | 12px | 700 | 1.2 |
| Body | 14px | 400 | 1.5 |
| Row Title / Control | 16px | 700 | 1.35 |
| Section Heading | 24px | 700 | 1.2 |

Rules:
- Do not use viewport-scaled font sizes for new app workflow/debug surfaces.
- Labels are uppercase only where the existing app pattern already uses uppercase metadata labels.
- Long trace IDs and idempotency keys must use wrapping monospace-style rendering via CSS only if already available; otherwise use standard font with `overflow-wrap: anywhere`.
- Button text must fit at 320px viewport width. Prefer `Approve exact draft`, `Retry publish`, and `View snapshot` over longer labels.

## Interaction Contract

### Approval Snapshot

- Approval action label: `Approve exact draft`.
- The approval button is disabled until one specific platform draft version is selected.
- Confirmation copy: `Approve {platform} draft v{version}? LocalPilot will freeze this exact payload for publishing. Later edits will create a new version.`
- After approval, show a frozen snapshot panel with a `View snapshot` affordance and the immutable fields listed in the Phase Surface Contract.
- Later edits/regeneration must visibly create a newer unapproved version without mutating the approved snapshot.

### Timeline

- Each platform has a visible timeline even before approval.
- The current state is visually dominant; completed states are subdued but readable.
- Terminal states are `published`, `failed`, and `manual_fallback_required`.
- Timeline rows must include timestamp, actor/system source, attempt number when applicable, and a one-line event summary.
- Empty event history must show the empty state copy defined below.

### Retry

- Retry is unavailable for `draft`, `needs_review`, `approved`, `queued`, `publishing`, and `published`.
- Retry button appears for `failed` and `retry_needed` only.
- Confirmation copy: `Retry this publish job? LocalPilot will create attempt #{nextAttempt} for the approved draft snapshot. Idempotency prevents duplicate published outcomes.`
- On confirm, the UI immediately shows a pending/queued attempt row and disables the retry button until the API returns the next state.
- Retry failures must preserve the prior attempt rows and append the new failure; never overwrite attempt history.

### Loading And Error States

- Initial `/app` API load: use a compact skeleton or muted row placeholders in the existing panel layout. Do not replace the whole app with a blank full-page loader.
- API write in progress: disable only the affected control and show a nearby inline pending label; keep navigation usable.
- API read failure: show an inline panel with the error copy below and a `Reload workflow` secondary action.
- API write failure: keep the user's selected draft visible, show the failed action message, and leave retry/reload controls explicit.

### Hidden Debug Route

- Route: `/app/debug`.
- Access pattern: direct URL only in Phase 1; do not add it to normal merchant sidebar navigation.
- Layout: compact operations table first, details drawer/panel second.
- Table columns: merchant, platform, job status, attempt count, latest trace ID, error class, updated timestamp, next action.
- Detail panel sections: approval snapshot summary, attempts, append-only events, redacted diagnostics.
- Use `Copy trace ID` for support handoff. Do not include copy buttons for redacted provider payloads.

## Copywriting Contract

| Element | Copy |
|---------|------|
| Primary CTA | Approve exact draft |
| Secondary publish CTA | Queue fake publish |
| Retry CTA | Retry publish |
| Snapshot affordance | View snapshot |
| Reload affordance | Reload workflow |
| Empty state heading | No publish jobs yet |
| Empty state body | Approve a platform draft to create a backend-backed publish job and timeline. |
| Error state | Publishing status could not load. Reload the workflow; if it fails again, share the trace ID with support. |
| Retry confirmation | Retry publish: LocalPilot will create a new attempt for the approved snapshot and use idempotency to avoid duplicate published outcomes. |
| Destructive confirmation | Reset demo workspace: This clears local demo preferences and reloads seeded backend records. Published audit records stay unchanged. |

Tone rules:
- Merchant-facing copy must be concrete and calm: say what happened and what to do next.
- Operator-facing copy may use terms such as trace ID, attempt, error class, idempotency key, and redacted diagnostics.
- Do not imply real Facebook or TikTok publishing in Phase 1. Use `fake publish`, `simulated provider`, or `backend-backed workflow` where needed.

## Responsive Contract

| Viewport | Contract |
|----------|----------|
| Desktop 1180px+ | Keep sidebar plus main grid. Timeline and snapshot may sit side by side when width allows. Debug route shows table plus details panel. |
| Tablet 768px-1179px | Collapse app grid to one column. Timeline stays full width. Snapshot follows selected draft details. Debug detail panel stacks below table. |
| Mobile 320px-767px | Sidebar follows existing mobile behavior. Platform tabs scroll or stack without shrinking labels below readable size. Timeline becomes vertical rows. Debug table becomes row cards with the same fields. |

Text fitting:
- No lifecycle label, CTA, trace ID, or error class may overflow its container.
- Long provider-safe diagnostic values wrap with `overflow-wrap: anywhere`.
- Timeline step labels stay visible; do not replace them with unexplained color-only dots.

---

## Accessibility Contract

- All status changes must have text labels; color is never the only signal.
- Approval and retry confirmation dialogs must be keyboard reachable, dismissible, and have a clear cancel path.
- `aria-current="step"` or equivalent semantic labeling should mark the active lifecycle state.
- Debug tables must use real table semantics when implemented as tables; mobile row-card fallbacks must preserve field labels.
- Buttons that open snapshot/debug details must declare the target with accessible text such as `View Facebook draft v3 snapshot`.
- Loading and write-pending states must not trap focus or remove the selected draft content from the DOM.
