<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Retry endpoint | Merchant requests another fake attempt for a failed job. |
| Retry service to idempotency records | Backend decides whether a retry can produce a terminal outcome. |
| Diagnostics to persistence and API | Provider-like diagnostics are sanitized before storage and response serialization. |
| Supply chain | Retry and redaction code uses existing local dependencies only. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-05-S | Spoofing | Retry endpoint | mitigate | Retry uses backend job ID and approval snapshot already stored in SQLite; client cannot submit a new provider target or alternate draft payload. |
| T-01-05-T | Tampering | Attempt history | mitigate | Attempts are insert-only; retry appends attempt number `n+1` and leaves prior attempts/events unchanged. |
| T-01-05-R | Repudiation | Retry audit trail | mitigate | Store retry actor/source, attempt number, trace ID, timestamps, status, and next action for each retry. |
| T-01-05-I | Information Disclosure | Redacted diagnostics | mitigate | Redaction runs before persistence and before serialization; tests inject token-like keys and fail on leakage in API output or UI-facing payloads. |
| T-01-05-D | Denial of Service | Repeated retry calls | mitigate | Allow retry only from `failed` and `retry_needed`; reject retry while publishing or after published; bounded fake adapter responses. |
| T-01-05-E | Elevation of Privilege | Retry outcome creation | mitigate | Idempotency prevents duplicate successful external outcomes for the same approval snapshot and connected channel. |
| T-01-SC | Tampering | npm/pip/cargo installs | mitigate | No package-manager installs; verify dependency lists remain unchanged. |
</threat_model>
